ZERO DOWNTIME — CLEAR EXPLANATION
🎤 SHORT INTERVIEW ANSWER (FIRST SAY THIS)

“Zero downtime deployment in Kubernetes is achieved using rolling updates, readiness probes, and multiple replicas to ensure that the application remains available while new versions are deployed.”

🧠 NOW EXPLAIN IN DETAIL (THIS IS WHAT THEY EXPECT)
🧩 1️⃣ REPLICAS (HIGH AVAILABILITY)
replicas: 3

👉 Meaning:

3 pods running at the same time
Even if 1 pod goes down → others serve traffic
🎯 ROLE

✅ Ensures application is always available
✅ Prevents downtime

🧩 2️⃣ READINESS PROBE (MOST IMPORTANT)
readinessProbe:
  httpGet:
    path: /
    port: 5000
🎯 ROLE

👉 Kubernetes checks:

“Is this pod ready to serve traffic?”

🔥 IMPORTANT BEHAVIOR
If NOT ready → no traffic sent
If ready → added to load balancer
💥 THIS IS KEY FOR ZERO DOWNTIME

👉 New pod is started
👉 Traffic is NOT sent until ready

🧩 3️⃣ LIVENESS PROBE
livenessProbe:
  httpGet:
    path: /
    port: 5000
🎯 ROLE

👉 Checks:

“Is application alive?”

🔥 IF FAILS
Pod restarted automatically
🧩 4️⃣ ROLLING UPDATE STRATEGY (CORE)
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 0
    maxSurge: 1
🧠 BREAKDOWN (VERY IMPORTANT)
🔹 maxUnavailable: 0

👉 Meaning:

❗ No existing pod will be stopped until new pod is ready

🔹 maxSurge: 1

👉 Meaning:

Kubernetes can create 1 extra pod during deployment

🔥 WHAT HAPPENS STEP-BY-STEP

Let’s say:

Current pods: 3 (v1)
New version: v2
🚀 STEP 1

👉 Kubernetes creates 1 new pod (v2)

Old: 3 (v1)
New: 1 (v2)
Total: 4 pods
🚀 STEP 2

👉 Waits until new pod is READY (readiness probe)

🚀 STEP 3

👉 Stops 1 old pod

Old: 2
New: 1
🔁 REPEAT

Until all pods are updated

🎯 FINAL RESULT

👉 No downtime
👉 No traffic interruption

🧠 COMPLETE FLOW
New Pod Created → Readiness Check → Traffic Shift → Old Pod Terminated
💥 CRITICAL POINT (INTERVIEW GOLD)

👉 Zero downtime is NOT only strategy

It depends on:

✅ 1. Replicas > 1
✅ 2. Readiness Probe
✅ 3. Rolling Update Strategy
✅ 4. Enough resources
🎤 FINAL STRONG ANSWER (USE THIS)

“We achieve zero downtime using multiple replicas and rolling update strategy. With maxUnavailable set to 0, no existing pods are terminated until new pods are ready. Readiness probes ensure that traffic is only routed to healthy pods, and maxSurge allows temporary scaling during deployment. This ensures seamless deployment without impacting users.”

⚠️ COMMON MISTAKES (YOU SHOULD KNOW)
❌ No readiness probe

👉 Traffic goes to unhealthy pod → downtime

❌ replicas = 1

👉 Downtime guaranteed

❌ maxUnavailable > 0

👉 Some downtime possible

❤️ FINAL UNDERSTANDING

👉 Kubernetes guarantees zero downtime ONLY IF:

Proper probes ✅
Correct strategy ✅
Enough replicas ✅
